# 판정 — KISA 홈페이지 취약점 진단·제거 가이드 21개 항목

기록일: 2026-07-17 · 원문: 한국인터넷진흥원(KISA) 「정보시스템 개발·운영자를 위한 홈페이지 취약점 진단·제거 가이드」(2013.12 작성/2014.1 발행, 84p) · 실측 기반, 짐작 없음

OWASP Top 10 감사([base/판정-보안기준.md](판정-보안기준.md))에서 이미 조치한 항목은 여기서 상호 참조만 하고 재작업하지 않는다. 21개 = 웹 어플리케이션 취약점 17개 + 웹 서버 취약점 4개.

## 우리 사이트 구조 (판정의 전제)

정적 Astro 사이트(빌드 시점에 전부 HTML로 굳음) + Cloudflare Pages 호스팅(서버 사이드 실행 없음) + DB 없음(콘텐츠는 git 저장소의 markdown) + Sveltia CMS(브라우저에서 GitHub API에 직접 통신, 우리 서버를 안 거침) + GitHub OAuth(인증은 전부 GitHub에 위임) + cms-auth-worker(OAuth 코드 교환만 담당하는 Cloudflare Worker). 이 구조 자체가 21개 항목 중 상당수를 "해당없음"으로 만든다 — 공격 표면이 되는 서버 사이드 코드·DB·자체 인증 로직이 원천적으로 없기 때문.

## 제2장 — 웹 어플리케이션 취약점 (17개)

| # | 항목명 | 개요 | 우리 상태 | 근거 |
|---|---|---|---|---|
| 1 | 운영체제 명령 실행 | 사용자 입력을 시스템 명령어 실행 함수에 전달해 OS 명령을 실행시킴 | **해당없음** | 서버 사이드 실행 코드 자체가 없음(정적 사이트, `exec`/`system` 류 호출 없음). Worker(cms-auth-worker)도 GitHub API 호출만, OS 명령 실행 없음 |
| 2 | SQL 인젝션 | SQL 질의문 필터링 미비로 DB 열람·조작 | **해당없음** | DB가 없음. 콘텐츠는 git 저장소의 markdown 파일(`src/content/`) |
| 3 | XPath 인젝션 | XPath/XQuery 질의문 조작으로 인증 우회·데이터 열람 | **해당없음** | XML/XPath 질의를 사용하는 기능 없음 |
| 4 | 정보누출 | 주석·에러 페이지로 민감정보·서버 경로 노출 | **조치완료** | 실측: `dist` HTML 전수 스캔(`TODO`/`password`/`secret`/민감주석 패턴) 매치 0건. 경로 순회 시도(`../../../etc/passwd`, URL 인코딩)에도 커스텀 404 또는 Cloudflare 자체 400만 반환, 서버 버전·경로 미노출(`Server: cloudflare`만, 버전 정보 없음) |
| 5 | 악성콘텐츠 | 입력값에 외부 악성코드 호출 스크립트 삽입 | **조치완료** | rehype-sanitize로 소식 body markdown의 `<script>`/`<iframe>` 삽입 차단 — 실측(OWASP A03, [판정-보안기준.md](판정-보안기준.md)) |
| 6 | 크로스 사이트 스크립트(XSS) | 게시판·URL에 악성 스크립트 삽입 | **조치완료** | 5번과 동일 방어(rehype-sanitize) + Astro 템플릿 자동 이스케이프(`{}` 보간은 기본 이스케이프, `set:html` 미사용) + CSP `script-src` 제한 |
| 7 | 약한 문자열 강도 | 회원가입 시 안전한 패스워드 규칙 미적용 | **해당없음** | 자체 회원가입/로그인 시스템 없음. CMS 인증은 GitHub OAuth로 완전 위임 — 비밀번호 정책은 GitHub이 관리 |
| 8 | 불충분한 인증 및 인가 | 인증 미흡·관리자 페이지 접근 제한 미비 | **위임 확인** | 이 저장소 코드에는 인증/인가 로직이 없음(전부 GitHub 위임). `/admin` 자체의 노출 여부는 아래 웹서버 2번·[OWASP A01](판정-보안기준.md)에서 별도 판정(Cloudflare Access 보류 확정) |
| 9 | 취약한 패스워드 복구 | 비밀번호 찾기 시 인증 미흡 | **해당없음** | 비밀번호 찾기 기능 자체가 없음 |
| 10 | 불충분한 세션 관리 | 세션ID 예측 가능·타임아웃 과다 | **위임 확인·완화 적용** | 세션 = GitHub OAuth 토큰(암호학적 난수, 예측 불가). 저장 위치(localStorage) 위험도는 OWASP A07에서 이미 기록, CSP로 스크립트 실행 제한해 완화 |
| 11 | 크로스 사이트 리퀘스트 변조(CSRF) | 정상/비정상 요청 구분 못해 의도치 않은 요청 실행 | **조치완료** | `cms-auth-worker/src/vendor.js` 소스 직접 확인: `/auth`에서 난수 `state`를 GitHub 요청 파라미터 + HttpOnly·Secure·SameSite=Lax 쿠키 양쪽에 심고, `/callback`에서 둘이 불일치하면 `CSRF_DETECTED`로 흐름 중단(이중 제출 쿠키 패턴, 정상 동작) |
| 12 | 자동화 공격 | 접근 횟수 제한 미설정으로 무차별대입·DoS 가능 | **부분 조치 (한계 문서화)** | Worker `/auth`·`/callback`에 Cloudflare Workers 네이티브 Rate Limiting(IP당 20회/60초) 적용·배포 완료. **단 실측 결과 한계 확인**: 같은 IP로 25회 연속 요청 시 Cloudflare 애니캐스트가 HKG/NRT/KIX 등 다른 콜로로 분산 라우팅해 콜로별 카운터가 각각 한도 밑이라 전부 통과(`wrangler tail` 로그로 직접 확인). Cloudflare 공식 문서도 "permissive, eventually consistent, intentionally designed to not be used as an accurate accounting system"이라 명시 — 하드 개런티 아닌 성긴 방어선. 이 Worker엔 자체 로그인 폼이 없어(자격증명 검사 자체를 안 함, GitHub이 전담) 고전적 무차별대입 시나리오가 원천적으로 성립하지 않는 점을 감안하면 비례성은 맞다고 판단. Durable Objects 등 정밀한 대안은 유료 플랜 필요성이 높아 미채택 |
| 13 | 파일 업로드 | 확장자 필터링 미비로 웹쉘 등 악성 파일 업로드 | **조치완료 (일부 잔여)** | `donation-reports.file` 필드에 `accept: 'application/pdf,.pdf'` 추가(이전엔 무제한). 잔여: 소식 body markdown 내 이미지 삽입은 Sveltia 자체에 확장자 제한 옵션이 없음(공식 문서 확인) — 다만 업로드 자체가 GitHub OAuth 인증된 편집자만 가능해(공개 익명 업로드 폼 없음) 고전적 위협모델(익명 공격자의 웹쉘 업로드)이 원천적으로 성립 안 함. 진짜 경계는 OAuth 토큰 보호(A07)이지 확장자 필터가 아님 |
| 14 | 경로추적 및 파일 다운로드 | 파라미터로 경로/파일명 처리해 임의 파일 다운로드 | **해당없음** | 파라미터 기반 다운로드 엔드포인트가 아예 없음. Cloudflare Pages는 빌드 시 정해진 정적 파일만 정확한 경로로 서빙 |
| 15 | 데이터 평문전송 | 로그인 시 민감데이터 평문 송수신 | **조치완료** | 전 트래픽 HTTPS + HSTS(`max-age=31536000; includeSubDomains; preload`, 실측 확인). 로그인 자체는 GitHub 도메인에서 GitHub의 HTTPS로 처리, 우리 사이트는 자격증명을 아예 취급 안 함 |
| 16 | 쿠키 변조 | 인증용 쿠키 변조로 권한 상승 | **해당없음/조치완료** | 본 사이트는 쿠키를 전혀 발급하지 않음(정적 사이트, 로그인 없음). Worker의 CSRF 쿠키는 HttpOnly+Secure+SameSite=Lax이고, 변조해도 `state` 불일치로 흐름이 중단될 뿐 권한 상승 경로가 없음(11번과 동일 근거) |
| 17 | URL/파라미터 변조 | 파라미터 조작으로 정보 유출·권한 상승 | **해당없음** | 파라미터 값으로 인가를 판단하는 로직이 없음(정적 라우팅, 사용자별 동적 페이지 없음) |

## 제3장 — 웹 서버 취약점 (4개)

| # | 항목명 | 개요 | 우리 상태 | 근거 |
|---|---|---|---|---|
| 1 | 디렉터리 인덱싱 | 서버 설정 미흡으로 강제 브라우징 시 파일 목록 노출 | **조치완료** | 실측: `/pagefind/`, `/_astro/`, `/uploads/`, 존재하지 않는 임의 디렉터리 등에 curl 직접 요청 — 전부 404, 목록 노출 없음(Cloudflare Pages 정적 호스팅 기본 동작) |
| 2 | 관리자페이지 노출 | 추측 가능한 경로로 관리자 페이지 접근·무차별대입 | **판정완료 (OWASP A01 참조)** | `/admin` 경로 자체는 추측 가능하나, 실질 방어는 GitHub OAuth(경로를 숨기는 건 근본 대책이 아님). Cloudflare Access(이중 인증 게이트) 도입은 보류 확정 — 사유: "이중 인증 부담 > 현재 위협 수준", 협업자 추가 시 재검토([판정-보안기준.md](판정-보안기준.md) A01) |
| 3 | 위치공개 | 백업·로그·임시파일이 웹에 노출 | **조치완료** | 실측: `dist` 전체에서 `.map`/`.env*`/`.bak`/`~`/`.old`/`.swp`/압축파일/`.sql` 검색 — 매치 0건. 소스맵 생성 안 함(Astro 기본 설정), 커밋 전 `.gitignore`로 생성물 관리 |
| 4 | 웹 서비스 메소드 설정 공격 | GET/POST 외 PUT/DELETE 등 허용 시 파일 조작 가능 | **조치완료** | 실측: 사이트 본체에 PUT/DELETE/OPTIONS/TRACE 요청 시 전부 405(Cloudflare Pages가 정적 자산에 GET/HEAD만 허용). Worker `/auth`는 소스에서 `method === 'GET'` 명시 확인, PUT/DELETE/POST 요청 시 전부 404 실측 확인 |
