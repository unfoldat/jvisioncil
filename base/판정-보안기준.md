## A05 — 보안 헤더 (Security Misconfiguration)

**수정 완료.** `scripts/inject-csp.mjs`가 `astro build` 직후 `dist/_headers`에 주입한다(`npm run build` = prebuild→build→inject-csp→verify 순서).

```
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Content-Security-Policy: default-src 'self'; script-src 'self' 'wasm-unsafe-eval' https://unpkg.com 'sha256-...'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: blob: https://*.githubusercontent.com; media-src blob:; frame-src blob:; connect-src 'self' blob: data: https://unpkg.com https://api.github.com https://www.githubstatus.com; worker-src 'self'; frame-ancestors 'none'; form-action 'self'; base-uri 'self'; object-src 'none'
```

### 왜 사이트 전체가 CSP 하나뿐인가 (판정)
Cloudflare Pages 공식 문서 확인: "An incoming request which matches multiple rules' URL patterns will inherit all rules' headers" — 겹치는 경로 규칙이 같은 헤더를 설정하면 **값을 콤마로 합친다(덮어쓰지 않는다)**. `/*`(사이트 전체, 엄격)와 `/admin/*`(Sveltia 전용, 느슨)에 CSP를 따로 두면 브라우저가 두 정책의 **교집합**을 강제해 admin이 깨진다. 그래서 공개 페이지 요구사항(Pagefind `wasm-unsafe-eval`/`worker-src`)과 admin 요구사항(Sveltia 공식 CSP 빌더 결과, GitHub 백엔드 선택 — sveltiacms.app/en/docs/security)의 **합집합**을 사이트 전체에 적용한다.
- 트레이드오프: 공개 페이지도 기술적으로 unpkg.com/githubusercontent.com을 허용하게 된다. 실사용 경로가 없어 위험은 낮지만, 완벽히 좁힌 상태는 아니라는 점을 기록해둔다. 페이지 파티션(경로별로 안 겹치는 블록 나열)으로 더 좁힐 수는 있으나, 새 페이지 추가 때마다 `_headers`를 안 고치면 조용히 CSP가 빠지는 위험이 더 커서 채택하지 않음.
- `script-src`의 해시는 `src/pages/검색.astro`의 유일한 인라인 스크립트(pagefind 동적 import, `@vite-ignore` 필요해 번들 분리 불가) 것. 빌드마다 자동 재계산되므로 스크립트를 고쳐도 수동으로 해시를 맞출 필요 없음.
- CSS는 `astro.config.mjs`의 `build.inlineStylesheets: 'never'`로 항상 외부 `.css` 파일로 뽑아, style-src에 사이트 자체 코드용 `'unsafe-inline'`이 필요 없게 만들었음(다만 admin 요구사항 때문에 결국 style-src에 unsafe-inline이 들어감 — 위 트레이드오프와 같은 이유).
- **배포 후 실측 완료**: `curl -sD -`로 라이브 응답 헤더 직접 확인(CSP·X-Frame-Options·HSTS 전부 적용됨 — 브라우저 `fetch()`는 이 중 일부를 JS에 안 보여주는 정상 동작이라 curl로 교차검증 필요했음). 홈/검색/`/admin` 실제 로드 시 콘솔 CSP 위반 0건, 검색 정상 작동, admin 설정 파싱·GitHub 로그인 진입까지 확인.

## A08 — 소프트웨어 무결성 (Sveltia CMS 로딩 방식)

**수정 완료 — CDN에서 자체 호스팅으로 전환.**
- 이전: `<script src="https://unpkg.com/@sveltia/cms@0.117.1/...">` — unpkg가 서빙하는 파일이 바뀌면(계정 탈취, CDN 캐시 오염 등) 우리 사이트가 그걸 그대로 실행하는 구조였음. 게다가 핀 버전(0.117.1)이 최신(0.171.1)보다 한참 뒤처져 있었음.
- 이후: `@sveltia/cms`를 npm 의존성으로 설치, `scripts/copy-cms.mjs`가 `npm run build`의 `prebuild` 단계에서 `node_modules/@sveltia/cms/dist/sveltia-cms.js`를 `public/admin/sveltia-cms.js`로 복사(커밋 안 함, `.gitignore` 처리, 매 빌드 재생성). `public/admin/index.html`은 이제 `./sveltia-cms.js`(같은 출처)를 로드.
- 남는 잔여 의존성: Sveltia 자체가 내부적으로 PDF.js·Prism(문법 강조) 등 보조 기능을 여전히 unpkg에서 가져올 수 있음(공식 문서 "About UNPKG origin" 확인) — 그래서 CSP의 `script-src`/`connect-src`에 `https://unpkg.com`이 남아 있음. 핵심 애플리케이션 번들 자체를 CDN에 의존하던 것(가장 큰 위험)은 제거됐고, 보조 라이브러리 fetch만 남음.
- SRI(Subresource Integrity) 해시는 채택 안 함 — 자체 호스팅이 SRI보다 근본적으로 나은 선택(가용성도 우리가 통제, 무결성도 npm의 lockfile+checksum이 보장)이라 SRI를 별도로 얹을 이유가 없어짐.

## A07 — 인증/세션 (OAuth 토큰 저장 위치)

**인지된 리스크 — Sveltia CMS 자체 설계에 따른 것. 완화책은 CSP로 대응. 위험도: 중간.**
Sveltia CMS 번들 소스에서 직접 확인(`node_modules/@sveltia/cms/dist/sveltia-cms.mjs`):
```
VD.account = {...VD.account, token: i, refreshToken: n}   // 토큰을 반응형 상태에 보관
XT.set('sveltia-cms.user', account)                        // 상태 변경 시 자동 저장
XT.set = (k,v) => globalThis.localStorage.setItem(k, JSON.stringify(v))
```
- GitHub OAuth 토큰(스코프 `repo,user` — sveltia-cms-auth Worker 소스에서 확인, [cms-auth-worker/src/index.js](../cms-auth-worker/src/index.js))이 평문 JSON으로 `localStorage['sveltia-cms.user']`에 들어간다.
- localStorage는 **origin 전체**에서 공유된다 — `/admin/` 경로뿐 아니라 사이트의 다른 어떤 페이지에서 스크립트가 실행되더라도 같은 origin이면 이 토큰을 읽을 수 있다. httpOnly 쿠키와 달리 JS로 직접 접근 가능.
- 토큰 스코프가 `repo,user`라 탈취되면 이 저장소뿐 아니라 계정이 접근 가능한 **모든 저장소**에 읽기/쓰기가 가능해진다 — 블라스트 반경이 큼.
- 완화책(이미 적용됨): ① A05의 CSP로 스크립트 실행 자체를 `'self' + 해시 + unpkg`로 제한 — 임의 스크립트 주입 경로를 좁힘. ② A03(마크다운 raw HTML) 이미 rehype-sanitize로 차단 — 알려진 XSS 벡터 제거.
- 잔여 위험(문서화만, 이번엔 조치 안 함): Sveltia는 PKCE 인증을 권장하지만(자체 문서 "What You Can Do") 지금 구성(sveltia-cms-auth Worker + 코드 교환 방식)은 PKCE가 아님. PKCE 전환은 Worker 재구성이 필요한 별도 작업 — 향후 과제로 남김.
- 토큰 탈취 의심 시 대응: GitHub → Settings → Applications → Authorized OAuth Apps에서 즉시 해당 앱 접근 취소(revoke) 가능.

## A01 — 접근 제어 (Cloudflare Access)

**판정 완료 — 지금은 안 건다.**
`/admin`은 GitHub OAuth 로그인 없이도 로그인 화면(정적 셸)까지는 누구나 도달 가능(공격 표면). 추가 층으로 Cloudflare Access(이메일 인증, 무료 티어 최대 50 유저)를 `/admin/*` 앞에 놓는 방안이 있었음.
- 보류 사유: **이중 인증 부담 > 현재 위협 수준.** GitHub OAuth로 이미 충분 — 로그인 화면 자체는 노출돼 있어도 실제 저장소 접근에는 GitHub 계정 인증이 필요하고, 지금 규모(독수리 단독 운영)에서 추가 이메일 인증 게이트는 얻는 방어 대비 매 접속마다의 마찰이 더 큼.
- 재검토 시점: 선생님(가현욱 교수 등 협업자) 계정으로 CMS 접근 권한을 이전·확장할 때 — 사용자가 늘면 공격 표면과 관리 필요성이 같이 커지므로 그때 다시 판단.

- **A08 무결성**: **해당없음.** 새 CDN·외부 스크립트 의존 추가 없음.
- **KISA 13(파일 업로드, [판정-KISA21항목.md](판정-KISA21항목.md) 참조)**: `image` 필드는 Sveltia `widget: image`(이미지 전용 피커) 사용 — `donation-reports.file`처럼 별도 `accept` 지정 불필요. 업로드 권한은 여전히 GitHub OAuth 인증된 편집자로 한정, 기존 판정(익명 공격자 위협모델 불성립)과 동일.

- **A05 설정 오류**: 해당없음. 새 서버·환경변수 없음.
- **데이터 무결성(신규 관점)**: `reference('network-orgs')`가 빌드 시점에 참조 유효성을 강제한다 — 존재하지 않는 기관을 가리키면 `astro build` 자체가 실패한다. 기존 자유 문자열 태그 방식(오타·존재하지 않는 기관명이 조용히 통과)보다 오히려 더 안전한 구조로 개선됨.
- **A08 무결성**: 해당없음. 새 외부 의존 없음.

**판정: 재검토 트리거 없음.**
